# Resource Detail Route Contract

## Purpose

Every ERP resource generated by the future Module Builder must expose a stable navigation contract for its primary screens:

- Index: `/{resource}`
- Create: `/{resource}/create`
- View/Detail: `/{resource}/{id}`
- Edit: `/{resource}/{id}/edit`

Warehouse is used as the canonical template module. The Warehouse detail screen is custom because it integrates detail fields, notes, attachments/media, and future timeline capabilities. For custom detail screens, top-level actions must not depend on fragile or guessed Vue route names.

## Current Decision

The Warehouse detail Edit action must navigate by canonical path:

```js
const editPath = computed(() => `/${resourceName}/${warehouseId.value}/edit`)

function goToEdit() {
  if (!warehouseId.value) {
    return
  }

  router.push(editPath.value)
}
```

This prevents accidental navigation back to the index view when a generated/custom route name does not match the actual route registry.

## Builder Rule

When generating a custom detail page:

1. Resolve `resourceName` from `Innoclapps.resourceName('<resource>')`.
2. Resolve the current record id from `route.params.id`.
3. Build canonical paths for detail and edit.
4. Use direct path navigation for critical actions unless the actual route name is discovered from the registered router.
5. Always verify:
   - detail opens `/{resource}/{id}`;
   - edit opens `/{resource}/{id}/edit`;
   - after saving edit, the app returns to either detail or index intentionally, not accidentally.

## Related Warehouse Capabilities

This route contract belongs to the base detail-page capability set and must be documented alongside:

- detail fields rendering;
- notes tab;
- attachments/media tab;
- attachments delete action;
- future timeline/changelog tab;
- future activities/calls/emails/documents capability toggles.
