# Builder Module Removal Strategy

Status: planning only
Date: 2026-07-02

## Purpose

Module removal is not one operation. The Builder must distinguish draft cleanup, non-destructive archive, UI hiding, disable, capability removal, uninstall, and destructive delete. This task documents the future strategy only; no runtime removal is implemented.

## Delete Draft Definition

Deleting a draft definition is safe only when the definition has never been published. The future implementation should check that no published module, generated runtime files, database migrations, permissions, menus, or runtime data depend on the definition.

Policy options:

- remove the active definition record
- remove or retain preview records according to retention settings
- keep an immutable audit record if compliance requires it
- delete derived preview artifacts only if they belong uniquely to that draft

## Archive Definition

Archiving is non-destructive. It hides a definition from active lists but keeps versions, previews, validation reports, and audit history. Archive should be available before destructive deletion because it preserves context and can support undo.

## Disable Published Module

Disable should be the safest first real lifecycle feature for published modules:

- module code remains installed
- database tables and business data are preserved
- menus and routes can be hidden or gated
- permissions remain auditable
- background jobs and automation should be paused
- search and RAG indexing should exclude disabled runtime surfaces unless explicitly allowed

Disable should be reversible and should not drop files or data.

## Hide Existing Module

Existing modules that should not be available in the app should be hidden through availability, config, permission, menu, or route gating where the architecture supports it. The Builder must not delete first-party or existing module code directly.

This operation needs a separate probe because existing modules may register menus, routes, permissions, policies, resources, and frontend routes in different places.

## Remove Capability

Removing a capability can affect generated artifacts, panels, tabs, actions, permissions, tables, files, media, comments, activities, imports, exports, search indexes, and RAG manifests. Future removal should start with disable/hide semantics and only delete data after dependency checks and explicit approval.

Capability removal should answer:

- which files/classes were generated by the capability?
- which data belongs to the capability?
- which other modules or records reference it?
- can the UI hide it while data is retained?
- does rollback need the removed capability restored?

## Uninstall Generated Module

Uninstall is safety-critical and should not be implemented before publish architecture exists. It requires:

- dependency graph
- publish manifest
- generated file manifest
- migration manifest
- data retention or destructive deletion decision
- backup and snapshot plan
- rollback strategy
- maintenance mode or staged execution plan if needed
- post-uninstall smoke checks

Uninstall must not infer what to delete from mutable current definition JSON.

## Destructive Delete

Destructive delete drops files, tables, columns, media, indexes, and data. It requires explicit future confirmation, backups, irreversible warnings, data retention rules, dependency analysis, and audit logging.

Destructive delete is not allowed in the current MVP.
