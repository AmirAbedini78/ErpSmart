# Resource Notes Integration

## Purpose

Notes integration connects a Resource record detail page to the shared `Notes` module. It must be implemented as a platform capability, not as a custom per-module note table.

For Warehouse, notes are used to store operational comments, follow-up context, and human-readable history around a warehouse master-data record.

## Required layers

A Resource can use Notes only when all of these layers are aligned:

```text
Domain Model relationship
Frontend detail tab
Core record-tab endpoint compatibility
Dependency on Notes module components
SmartDocs/RAG manifest update
```

## Backend model contract

The resource model must expose a morph-to-many relation named `notes`:

```php
public function notes(): MorphToMany
{
    return $this->morphToMany(Note::class, 'noteable')->withTimestamps();
}
```

This uses the shared `noteables` pivot maintained by the Notes module. The Builder must not create a separate `warehouse_notes` table unless the domain explicitly requires non-Core note semantics.

## Frontend detail tab contract

The detail view must import the Notes record-tab components:

```js
import RecordTabNote from '@/Notes/components/RecordTabNote.vue'
import RecordTabNotePanel from '@/Notes/components/RecordTabNotePanel.vue'
```

The detail view must also provide the shared resource synchronization functions used by record tab components:

```js
provide('fetchResource', fetchResource)
provide('synchronizeResource', synchronizeResource)
provide('detachResourceAssociations', detachResourceAssociations)
provide('incrementResourceCount', incrementResourceCount)
provide('decrementResourceCount', decrementResourceCount)
```

## Runtime behavior

`RecordTabNotePanel` loads notes through the generic resource record-tab request pattern:

```text
{resource.path}/notes?page=1&timeline=1
```

For Warehouse this means the runtime endpoint is expected to resolve to a resource path similar to:

```text
/api/warehouses/{id}/notes
```

The exact path is generated by the Core Resource engine and the `resource.path` value returned by the detail endpoint.

## Builder rule

When a module builder enables Notes for a generated module, it must generate:

```text
1. model morphToMany notes() relation
2. frontend detail tab components
3. provide() synchronization hooks in the detail view
4. module manifest capability: notes
5. validation checklist entry
6. history entry
```

## Validation checklist

1. Open a Warehouse detail page.
2. Confirm a Notes tab is visible.
3. Add a note.
4. Confirm the note appears without page reload.
5. Refresh the page and confirm the note is still visible.
6. Edit/delete the note if the Note policy permits it.
7. Confirm the browser console has no missing component or undefined injection errors.

## Known risks

- If the Notes backend files are missing or zero-byte in a local analysis zip, rely on the real installed project, not the analysis archive.
- If `resource.notes` or `resource.notes_count` is absent on the initial detail response, the Notes panel should still lazy-load data through the record-tab endpoint.
- If the detail view does not call `provide()`, note creation can fail because `NotesCreate.vue` injects synchronization callbacks.

## 2026-06-23 correction: Notes requires HasTimeline

The first Warehouse Notes integration generated the `notes()` morph relation and frontend tab wiring, but clicking the Notes tab returned 404. The Core route for record-tab timeline associations is:

```text
GET /api/{resource}/{resourceId}/{associatedResource}?timeline=1
```

This route is handled by `AssociationsController::show()`. When `timeline=1`, the controller aborts with 404 unless:

```text
Timelineables::hasTimeline($request->record()) === true
Timelineables::isTimelineable($associatedResource->newModel()) === true
```

For the resource record, `hasTimeline()` means the model must use:

```php
Modules\Core\Common\Timeline\HasTimeline
```

Therefore a Notes-enabled module requires:

```text
1. Model uses HasTimeline
2. Model defines notes() relation
3. Related Note model is Timelineable in the Notes module
4. Frontend detail page wires RecordTabNote and RecordTabNotePanel
5. Cache is cleared after adding the trait, because timeline subjects are cached forever
```

Recommended commands after adding `HasTimeline`:

```bash
php artisan optimize:clear
php artisan cache:clear
```

In Docker:

```bash
docker compose exec app php artisan optimize:clear
docker compose exec app php artisan cache:clear
docker compose restart app nginx
```

## 2026-06-24 Detail view stability fix

After adding Notes, the Warehouse detail page entered a heavy Vue update loop when rendering `DetailFields`.
The browser console showed repeated `FieldInlineEdit` render warnings and repeated update/flush cycles in `WarehousesView.vue`.

Root cause:

- The Warehouse API response is valid for display and Notes, but the generic Core `FieldInlineEdit` expects a complete edit contract on the resource object, including `authorizations` and edit metadata.
- A custom Resource detail page that renders `DetailFields` without normalizing this contract can crash before the Notes tab is usable.
- Missing i18n keys such as `warehouse::warehouse.back_to_warehouses` and `warehouse::warehouse.active` are not the crash root cause, but they must be fixed in the same step.

Fix:

- Normalize the detail resource before passing it to `DetailFields`, `RecordTabNote`, and `RecordTabNotePanel`.
- Add safe default `authorizations`, `_edit_disabled`, and `_sync_timestamp` values.
- Disable inline editing for Warehouse detail fields for now; users edit Warehouse records through the dedicated edit page.
- Add the missing Warehouse language keys.

Builder rule:

For a custom module detail page, never assume the generic Resource API response already contains the complete inline-edit contract. Either reuse the exact Core record view component contract, or normalize the resource and disable inline edit until the contract is fully supported.


## 2026-06-24 Notes path contract fix

`RecordTabNotePanel` uses Core `useRecordTab`. The default request is built from `config.resource.value.path` and `timelineRelation`. For Warehouse, the normalized resource must therefore include:

```js
path: `/warehouses/${id}`
notes: []
notes_count: 0
```

If `path` is missing, the request becomes `/api/undefined/notes?page=1&per_page=15&timeline=1`. This is a frontend contract issue, not a backend route or `HasTimeline` issue.

Builder rule: every generated record detail view that uses Core record tabs must normalize `resource.path` before rendering tab panels.


## 2026-06-24 Notes backend association contract fix

Core Notes integration is a two-sided relation contract:

1. The domain model must expose the forward relation, for example `Warehouse::notes()`.
2. The associated model must expose the inverse relation matching the domain resource `associateableName()`, for example `Note::warehouses()`.

Warehouse now registers the inverse relation from `WarehouseServiceProvider` using Eloquent `resolveRelationUsing`:

```php
Note::resolveRelationUsing('warehouses', function (Note $note): MorphToMany {
    return $note->morphedByMany(Warehouse::class, 'noteable')->withTimestamps();
});
```

Why: Core association/timeline requests such as `/api/warehouses/{id}/notes?timeline=1` and note creation via `POST /api/notes?via_resource=warehouses&via_resource_id={id}` need the Notes model to understand the `warehouses` relation. Without the inverse relation, listing notes can fail with 500 and creating notes can fail because the association payload cannot be synchronized correctly.

Builder rule: every generated Notes-enabled resource must generate both sides of the morph association, or register the inverse relation dynamically in the module service provider when the associated model belongs to another module.


## 2026-06-24 Notes POST validation contract

`GET /api/warehouses/{id}/notes?timeline=1` and `POST /api/notes?via_resource=warehouses&via_resource_id={id}` are different contracts.

For POST to pass, the generated module must provide all of the following:

1. Subject model forward relation, e.g. `Warehouse::notes()`.
2. Subject model timeline support, e.g. `HasTimeline`.
3. Associated model inverse relation, e.g. `Note::warehouses()`.
4. Subject resource association support, e.g. `WarehouseResource use AssociatesResources`.
5. Create-resource validation hook for Core Notes when the associated module is external to the Notes module.

If step 5 is missing, the API returns `422` with `The selected via resource is invalid.` even when the body is not empty.
