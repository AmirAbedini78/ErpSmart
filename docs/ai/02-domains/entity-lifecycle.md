# Entity Lifecycle — Concord/ErpSmart Resource Module Pattern

## Purpose
This document is the compact RAG map for creating or modifying first-class business entities in ErpSmart / Concord CRM.
It is written for humans and for small local LLMs such as Qwen Coder 1.5B. Do not treat an entity as a plain CRUD table.
A correct entity must pass through the Concord resource engine, table engine, module engine, frontend route registration, permissions, translations, and optional workflow/association layers.

## Placement
- File: `docs/ai/02-domains/entity-lifecycle.md`
- Type: domain / pattern document

## Mental model
A module contributes features into the core application. A Resource is the main bridge between backend model, API/table, frontend forms, permissions, menu, builder/custom fields, workflow, and UI panels.

```text
module.json
  -> Module ServiceProvider
    -> resources[] registration
    -> routes registration
    -> workflow triggers/actions registration
    -> settings/defaults/menu metrics/scriptData
      -> Resource class
        -> Model
        -> Table
        -> Fields
        -> Filters
        -> Actions
        -> Cards
        -> Menu item
        -> Settings menu item, if needed
        -> Panels/Tabs/relationships
          -> Frontend app.js/routes.js/components/views
```

## Entity lifecycle checklist
Every first-class entity must be checked against these steps.

### 1. Module registration
Required files:

```text
modules/{Module}/module.json
modules/{Module}/composer.json
modules/{Module}/app/Providers/{Module}ServiceProvider.php
modules/{Module}/app/Providers/RouteServiceProvider.php
```

Rules:
- `module.json.providers` must point to the module provider.
- Provider extends `Modules\Core\Support\ModuleServiceProvider` when the module registers resources or script data.
- Provider must register its RouteServiceProvider.
- Provider may expose `protected array $resources = [...]`.

### 2. Database layer
Required for persistent entity:

```text
modules/{Module}/database/migrations/*create_{table}_table.php
modules/{Module}/database/factories/{Entity}Factory.php      optional but recommended
modules/{Module}/database/seeders/*                          optional
modules/{Module}/database/state/*                            optional for default state
```

Rules:
- Respect project table prefix indirectly; write logical table names in migrations and models.
- If SaaS is enabled later, tables that should be tenant-aware may need `tenant_id`.
- Seeders/state must create required defaults. Example bug discovered: Workflows failed because `activity_types` was empty and `task` flag was missing.

### 3. Model layer
Required files:

```text
modules/{Module}/app/Models/{Entity}.php
modules/{Module}/app/Policies/{Entity}Policy.php
```

Rules:
- Model handles relationships and searchable/display attributes.
- Policy controls access for Resource actions/table/detail.
- If entity participates in shared features, use traits/contracts used by existing modules.

### 4. Resource layer
Required files:

```text
modules/{Module}/app/Resources/{Entity}.php
modules/{Module}/app/Resources/{Entity}Table.php
```

Resource responsibilities:
- bind model: `public static string $model`
- define title/name/icon/search/detail behavior
- define menu item
- define fields
- define table class
- define filters/actions/cards
- define panels/tabs/associations if related to other resources
- expose builder/custom-field support via proper Core contracts

Known reference:
`modules/Deals/app/Resources/Deal.php` implements many capabilities:

```text
AcceptsCustomFields
BillableResource
Exportable
Importable
Mediable
PipesComments
Tableable
WithResourceRoutes
```

Do not copy all contracts blindly. Choose only what the new entity actually needs.

### 5. Backend routes and controllers
Resource routes may be generated by the Core resource engine when Resource implements `WithResourceRoutes`.
Custom endpoints go into:

```text
modules/{Module}/routes/api.php
modules/{Module}/app/Http/Controllers/Api/*
modules/{Module}/app/Http/Resources/*Resource.php
```

Use custom controllers only for non-standard behavior such as boards, summaries, stage changes, status transitions, or bulk operations.

### 6. Frontend registration
Required files:

```text
modules/{Module}/resources/js/app.js
modules/{Module}/resources/js/routes.js
modules/{Module}/resources/js/views/{Entity}Index.vue
modules/{Module}/resources/js/views/{Entity}Create.vue
modules/{Module}/resources/js/views/{Entity}View.vue
modules/{Module}/resources/js/components/*
```

Rules:
- `resources/js/app.js` registers components and routes during `Innoclapps.booting`.
- `routes.js` returns Vue router entries.
- For settings pages, route is usually added under router name `settings` inside module `app.js`.
- Build/dev asset status matters: if `public/hot` exists, Laravel loads Vite dev assets from port 5173. If node is stopped, remove `public/hot` and build.
- For built-in/root-bundled modules, also import the module app file from `resources/js/app.js`, for example `import '@/Warehouse/app.js'`. If this is missed, Vue Router will not know the route and `/module-path` will show SPA 404 even when the backend Resource exists.

### 7. Translations
Required path:

```text
modules/{Module}/lang/en/*.php
```

Recommended keys:
- singular/plural labels
- field labels
- validation messages
- settings page titles
- menu labels

### 8. Settings and builder integration
Concord entities are usually not hard-coded only. They may use the builder/custom-field/table-view system.
A builder-compatible entity should be represented as a Resource and should use existing Core field/table mechanisms instead of custom one-off forms.

Minimum builder-aware check:
- Is Resource `fieldsCustomizable` enabled if needed?
- Does Resource implement `AcceptsCustomFields` if custom fields should be supported?
- Does Resource table support views if user-defined table views are needed?
- Are fields declared through Core Field classes rather than raw Vue-only form inputs?

### 9. Workflow integration
Workflow layer usually has:

```text
modules/{Module}/app/Workflow/Triggers/*
modules/{Module}/app/Workflow/Actions/*
```

Provider registers triggers through `Modules\Core\Workflow\Workflows::triggers([...])`.
Do not add workflow before the entity MVP is stable.

### 10. Associations and panels
Existing mature entities show associated panels such as:
- Deals connected to Companies/Contacts.
- Activities connected to resources.
- Documents/Notes/Emails filter by resource.

A new entity must decide associations explicitly:
- Can it have activities?
- Can it have documents?
- Can it be billable/product-related?
- Can it appear in global search?
- Can it be related to company/contact/deal?

### 11. Validation / smoke checks
After adding or modifying an entity:

```bash
php artisan module:list
php artisan route:list | grep -i {entity}
php artisan migrate --force
php artisan optimize:clear
npm run build
```

Browser checks:
- index page opens
- create form opens
- record can be created
- table displays record
- detail page opens
- settings route opens, if added
- no 500 in Network
- no frontend mount error such as `CreateApplication is not defined`

When asked to create or change an entity, read only these first:


Then inspect reference files from one mature module only, usually Deals or Contacts.
Never ask a small LLM to scan all modules at once.

## Known pitfalls
- `public/hot` owned by `www-data` can force dev asset loading even when node is stopped.
- `bootstrap/cache/*.php` can keep disabled modules booting until manually removed.
- SaaS module can mutate indexes during provider boot; keep it disabled until fixed.
- Telescope/Pulse tables must be excluded from tenant schema mutation.
- Missing seed data can break workflow APIs. Example: missing `activity_types.task` caused Workflow screen server error.
- Missing frontend import can break new entity pages. Example: Warehouse backend Resource existed as `warehouses`, but `/warehouses` still showed 404 until `resources/js/app.js` imported `@/Warehouse/app.js`.
